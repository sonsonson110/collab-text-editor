# Phase 1: Decouple with Redis Pub/Sub

## Overview
Currently, the `api-server` and `sync-server` are coupled via direct HTTP calls. When the `api-server` mutates room permissions (e.g., role updates, member removals, access mode changes), it notifies the `sync-server` by sending a POST request to the internal HTTP server running on `PORT + 1` (default `1235`).

This phase proposes replacing these point-to-point HTTP notifications with a Redis Pub/Sub mechanism.

## Proposed Architecture
- **Redis Pub/Sub**: Introduce Redis as the message broker between the two services.
- **api-server (Publisher)**: Update `SyncServerNotifier` to publish `PermissionEvent` payloads to a Redis channel (e.g., `room-permissions`).
- **sync-server (Subscriber)**: Remove the internal HTTP server (`http.createServer`) from `src/index.ts`. Instead, connect to Redis and subscribe to the `room-permissions` channel.

## Audit & Implementation Details

### 1. `api-server` Changes
- Add `spring-boot-starter-data-redis` dependency.
- Configure Redis connection properties in `application.yml`.
- Refactor `SyncServerNotifier.java`:
  - Instead of using `RestTemplate` to POST to the internal sync-server URL, use `RedisTemplate` to execute a `.convertAndSend("room-permissions", payload)`.
  - The `@Async` annotation might no longer be strictly necessary since Redis pub/sub operations are generally fast and non-blocking, but keeping it ensures the web thread is completely free from network overhead.

### 2. `sync-server` Changes
- Add `ioredis` dependency.
- In `src/index.ts`:
  - Delete the `internalServer` HTTP setup block.
  - Delete `src/api/permissionHandler.ts`'s HTTP `createInternalHttpHandler` and replace it with a Redis message handler.
  - Initialize an `ioredis` subscriber client.
  - Subscribe to `room-permissions`. When a message arrives, parse it and execute the existing permission fan-out logic (disconnecting blocked users, broadcasting `MSG_PERMISSION_CHANGED` to allowed users).

## Advantages
- **Horizontal Scalability**: We can now spin up multiple `sync-server` instances. When the `api-server` publishes a permission change, ALL `sync-server` nodes will receive the message and apply the permission updates to their locally connected WebSockets. 
- **Decoupling**: The `api-server` no longer needs to know the IP address or port of the `sync-server`.
- **Security**: Removes the need for the `INTERNAL_API_SECRET` and exposing an internal HTTP port, reducing the attack surface.

## Risks & Mitigations
- **Message Delivery Guarantee**: Redis Pub/Sub is fire-and-forget (at-most-once delivery). If a `sync-server` node drops connection to Redis briefly, it might miss a permission event. 
  - *Mitigation*: Since WebSocket clients also drop and reconnect on network failures, and new connections re-fetch their `effectiveRole` via JWTs generated by the `api-server` from the database, the system will eventually self-correct. However, for strict security, we might consider Redis Streams instead of Pub/Sub if we need guaranteed delivery and history.
